How repo-genesis Works
01

From Idea to Repo

You hand it one sentence. It hands you back a GitHub project with a repo, formal requirements, issues, and milestones — with receipts for everything.

One sentence in, a whole project out

repo-genesis is an agent you run from the command line. You give it a raw idea, and it bootstraps a complete GitHub repository around it: researched, specified, and broken into tracked work items.

YOU TYPE 

cd code
uv run python flow.py "When was Claude Shannon born and what did he contribute to information theory?"
PLAIN ENGLISH

Go into the folder where the agent lives...

...and start it with a goal written exactly the way you would say it to a person. No forms, no special syntax — the sentence IS the input.

Think of it as a city planning office. You walk in with a rough sketch of a building. The office checks your permit to build (are you logged in?), surveys comparable buildings nearby, drafts formal blueprints, and issues numbered permits for each piece of work — and every permit cites the blueprint line that justifies it.

The eight steps of a run

Every run walks the same arc — from checking your identity to publishing the plan and replaying what happened:

1
Verify

Confirm you are logged into GitHub. If not, stop cleanly — never act half-authenticated.

2
Bootstrap

Create the repository with a license and a README holding the initial idea.

3
Research

Search GitHub for 3–5 similar projects and read each one's README on the live page.

4
Compare & refine

Build a comparison table of those projects and sharpen the idea against the evidence.

5
Specify

Write formal requirements, each with a stable ID like REQ-001 that can be cited later.

6
Publish

Open one GitHub issue per piece of work, each traced to its requirement, grouped into milestones.

7
Review

A QA critic checks the chain: every requirement has an issue, every issue has a milestone.

8
Replay

Emit a step-by-step record of everything it did, every page it visited, and what it cost.

💡
Traceability is the whole point

Every issue cites a requirement; every requirement cites the refined idea; the refined idea cites real projects it was compared against. Engineers call this a traceability chain — it means no work item exists without a reason on record.

Two machines under the hood

The codebase splits into two independent programs that talk over the network:

🧠
The agent (code/)

Plans the work, runs the skills, drives the browser, and remembers what it learned. This is the brain.

🚪
The gateway (gateway/)

One front door for every LLM call. It picks a provider, respects rate limits, and writes every call into a cost ledger.

📦
The paper trail (code/state/sessions/)

Every run leaves artifacts on disk: the plan graph, each step's exact prompt and result, and browser screenshots.

And here is the very first thing the agent does with your sentence — it writes it down, then schedules exactly one worker, the Planner:

graph = Graph()
graph.add_node("planner", inputs=["USER_QUERY"])

Save your question to disk first, so the run can be resumed or replayed later.

02

The Growing Graph

The plan is not written once and executed. It is a living map that draws new roads while the agent is already driving.

A detective’s evidence board

Picture a detective’s evidence board: photos pinned to a wall, strings connecting them. When a new lead comes in, the detective pins a new photo and runs new strings to it. The board grows as the case unfolds — nobody knew its final shape on day one.

That board is the agent’s plan. Each pinned photo is a node (one task, run by one skill), and the strings are arrows saying “this task feeds that one.” Engineers call this shape a DAG. The orchestrator in code/flow.py is the detective: it never investigates anything itself, it just decides which leads are ready to chase and pins what comes back.

🧩
Why a graph and not a checklist?

A checklist runs one item at a time. A graph knows which tasks don’t depend on each other — so three research tasks can run at the same time, and the comparison task waits for all three. Same work, much faster.

The heartbeat: run what’s ready, repeat

The whole engine is one loop. Every beat, it asks: which tasks have all their inputs finished? Run those. Pin the results. Ask again.

CODE — code/flow.py 

while True:
    ready = graph.ready_nodes()
    if not ready and not graph.has_running():
        break
PLAIN ENGLISH

Keep going around this loop forever (until told to stop)...

Ask the board: which tasks have every upstream task finished?

If nothing is ready AND nothing is still running...

...the case is closed. Stop the loop.

And here is the rule that decides “ready” — a task can start when every task feeding it is either done or deliberately skipped:

CODE — code/flow.py 

preds = list(self.g.predecessors(nid))
if all(self.g.nodes[p]["status"] in ("complete", "skipped") for p in preds):
    out.append(nid)
PLAIN ENGLISH

Gather every task whose string points INTO this one — its suppliers.

If every single supplier has either finished or been waved off...

...this task goes on the “run it now” list.

Five ways new pins land on the board

The graph starts with a single node — the Planner — and grows through exactly five mechanisms:

🌱

The Planner’s seed plan

The Planner reads your goal and pins the first batch of tasks — the opening leads of the case.

➕

Dynamic successors

Any skill can finish its work and say “now someone should do THIS next” — pinning follow-up tasks itself.

🔗

Static successors

Some pairs are wired in config: whenever the Coder finishes, a sandbox run is automatically pinned after it.

🔍

Critic auto-insertion

Skills marked critic: true get a quality-check node spliced onto every outgoing string before anyone uses their output.

🛠️

Recovery re-planning

When a task fails, the Planner is called back with a failure report to draw a detour around the dead end.

💡
Growth needs a leash

A plan that can extend itself can also run away with itself. One line guards the whole system: MAX_NODES = 60 — a hard cap so a looping Planner cannot grow the board forever. Whenever you build something self-extending, give it a leash first.

03

The Skill Catalogue

Every worker the agent can hire is defined by two small files — and adding a new one never touches the engine.

A binder of laminated job cards

Imagine a staffing binder. Each page is one laminated job card: the role’s name, what it does, which equipment it may use, and how much creative freedom it gets. To hire a new kind of worker, you add a page to the binder — you don’t rebuild the office.

In this codebase, a skill is exactly that: one entry in code/agent_config.yaml (the job card) plus one prompt file in code/prompts/ (the job briefing). There is no Python class per skill — the engine reads the binder and hires accordingly.

CODE — code/agent_config.yaml 

critic:
  prompt: prompts/critic.md
  temperature: 0.0         # pass/fail; we want determinism
  max_tokens: 500
  description: Evaluates an upstream node's output; emits pass or fail with rationale.
PLAIN ENGLISH

A worker called “critic” exists.

Its job briefing lives in this text file.

Zero creative freedom — a judge should give the same verdict every time.

Keep its answers short (about 500 words’ worth).

Its job: check another worker’s output and say pass or fail, with a reason.

Ten cards in the binder

The catalogue ships with ten skills. Each one is a specialist, not a generalist:

planner Breaks your goal into the task graph; redraws it when something fails
researcher Searches the web and fetches pages — the only skill allowed those tools
browser Drives a real web browser through a four-layer cascade (next module!)
retriever Searches the agent’s own indexed knowledge before going to the web
distiller Extracts structured facts from raw text — always followed by a Critic check
summariser Condenses long content into short form
critic Judges another node’s output: pass or fail, with rationale
coder Writes Python code — automatically followed by the sandbox runner
sandbox_executor Runs the Coder’s code in a sandbox and reports what happened
formatter Writes the final answer the user reads — the conventional last node

The engine reads the binder once

At startup, the registry loads every card. This is the entire “hiring system”:

CODE — code/skills.py 

class SkillRegistry:
    def __init__(self):
        cfg = yaml.safe_load(AGENT_CONFIG_PATH.read_text(encoding="utf-8"))
        self._skills: dict[str, Skill] = {n: Skill(n, c) for n, c in cfg.items()}
PLAIN ENGLISH

The registry is the office’s staffing desk.

When the office opens for the day...

...it reads the whole binder file from disk...

...and turns every page into a ready-to-hire worker, looked up by name.

Three optional fields on a job card quietly drive the orchestrator behaviours you met in Module 2:

tools_allowed Which equipment this worker may use (web search, page fetch, knowledge search) — empty means words only
internal_successors Who is ALWAYS hired right after this worker finishes (Coder → sandbox runner)
critic: true Every hand-off from this worker gets a quality inspector spliced in first (Distiller)

Where you plug in — and what you never touch

🔌
Extending = adding pages, never rewiring

To add a brand-new capability: write one prompt file in code/prompts/ and one entry in code/agent_config.yaml. The Planner learns it exists from its own prompt, and the orchestrator can run it immediately. That’s the whole procedure.

⛔
The orchestrator is off-limits

This project has a standing rule: new features plug in through the catalogue; code/flow.py stays unmodified. If a change you’re planning requires editing the engine, that’s the signal to redesign it as a skill instead. This keeps one battle-tested engine running many evolving skills.

💡
This pattern has a name

Behaviour defined in data files instead of code is called configuration-driven design. When you ask an AI assistant to “make the new behaviour config-driven,” this is exactly what you’re asking for.

04

The Browser Cascade

Reading a web page can cost almost nothing — or a vision AI looking at screenshots. The skill always tries the cheap way first.

Getting answers out of a building

Suppose you need information from an office building. You’d read the sign outside first. If that’s not enough, follow the posted directions. Still stuck? Ask at the front desk, where the receptionist reads the building directory to you. Only as a last resort do you walk the halls yourself, looking around room by room.

The Browser skill (code/browser/) treats every web page the same way. Many modern pages are JavaScript-rendered or hide their answers behind filters and buttons, so the Researcher’s simple page-fetch comes back empty. This skill escalates through four layers, stopping at the first one that succeeds:

📋

1 · Extract

Download the page and pull out the readable text. No browser, no AI — nearly free.

🎲

2a · Deterministic

If the caller supplied exact click/fill instructions, replay them in a real browser. No guessing allowed.

🗣️

2b · A11y

A real browser reads the page’s accessibility tree and a text-only AI decides what to click, step by step.

👀

3 · Vision

Screenshot the page, draw numbered tags on every clickable element, and let a vision model literally look.

Watch one request climb the ladder

The goal: “filter laptops under ₹80,000 and extract the top 3.” Step through what the cascade does:

🎯
Goal + URL
📋
Extract
🗣️
A11y
👀
Vision
Click “Next Step” to begin
💰
Cheap first, smart later

This is an escalation ladder — a pattern worth requesting by name when you steer an AI assistant: “try the cheap path first and escalate only on failure.” Most pages never get past layer 1, so the average cost stays tiny.

The rule that triggers escalation

How does layer 1 know its answer is not good enough? Two checks — too little text, or a goal that demands interaction:

CODE — code/browser/skill.py 

if len(content) < 200:
    return False
interactive_verbs = ("click", "fill", "select", "type", "drag",
                     "filter", "sort", "submit", "navigate")
if any(v in goal.lower() for v in interactive_verbs):
    return False
return True
PLAIN ENGLISH

Got fewer than ~200 characters of text? That’s a husk, not a page. Not useful.

Here is a list of action words...

...that all mean “you must DO something on the page.”

If the goal asks for any of them, reading alone can never succeed. Not useful.

Otherwise — the cheap layer’s answer stands. Done.

🚧
Blocked is a verdict, not a defeat

If any layer finds a CAPTCHA or a login wall, the skill does NOT keep climbing — clicking harder at a wall just wastes money. It returns immediately with the code gateway_blocked, and the Planner is recalled to route around the blocked site. Escalation fixes “too hard”; it cannot fix “not allowed.”

05

The Gateway

Seven AI providers, one front door. Every call gets routed, rate-limited, retried — and written into a ledger.

An air traffic control tower

The agent never calls an AI provider directly. Every request flies through the Gateway (gateway/main.py) — a control tower with seven runways: Gemini, Groq, Cerebras, NVIDIA, OpenRouter, GitHub Models, and a local Ollama.

The tower’s job, on every single call:

🛣️
Assign a runway

Pick a provider that has the needed capabilities (tools, vision, structured output) and free capacity right now.

⏳
Respect the congestion rules

Each provider has a rate limit. The tower tracks calls per minute, per day, and tokens spent, and never sends a plane to a full runway.

🔁
Reroute on trouble

Provider down or busy? The call falls through a failover ladder to the next one. The caller never notices.

📑
Log every landing

Every call — tokens, latency, which skill made it, which session — lands in a SQLite ledger. Cost accounting comes free.

Who flies on which airline

Each skill is pinned to a preferred provider in one tiny file — matching the job to the model’s strengths:

CODE — gateway/agent_routing.yaml 

planner: gemini
researcher: gemini
distiller: gemini
summariser: gemini
critic: groq
formatter: gemini
retriever: github
sandbox_executor: github
coder: gemini
browser: gemini
PLAIN ENGLISH

The Planner needs the strongest reasoning — give it Gemini.

The bulky reading-and-writing skills also go to Gemini, which handles very long documents comfortably.

The Critic only says pass or fail — send it to Groq, a small, fast, cheap worker.

Classifier-ish skills ride the free GitHub Models tier.

Changing any of this is a one-line edit. No code.

📌
A pin is a preference, not a promise

If a pinned provider is cooling down, the gateway waits briefly, then falls back through the normal ladder. The system degrades gracefully instead of refusing to fly.

How a runway says “not now”

Before any call, the router asks the provider’s bookkeeping: can I use you right now? Three of its checks:

CODE — gateway/router.py 

wait = limits["cooldown"] - (now - self.last_call)
if wait > 0:
    return False, f"cooldown ({wait:.1f}s)"
if len(self.calls_minute) >= limits["rpm"]:
    return False, "RPM limit"
if self.calls_today >= limits["rpd"]:
    return False, "RPD limit"
PLAIN ENGLISH

Has enough time passed since this provider’s last call? Some need a short breather between requests.

If not — refuse, and say exactly how long the breather has left.

Already at the requests-per-minute ceiling? Refuse.

Already at the requests-per-day ceiling? Refuse.

Every refusal names its reason — so the router can report exactly why it skipped a provider.

Beyond chat, the same front door serves the rest of the system:

/v1/chat The main channel — every skill’s text and tool-use calls
/v1/vision Screenshot questions — what the Browser’s layer 3 calls
/v1/embed Turns text into vectors for the agent’s memory search
/v1/cost/by_agent The ledger, sliced: what each skill spent, per session

06

When Things Break

Failures are not exceptions here — they are inputs. The system triages every one and keeps a black box of the whole flight.

The triage desk

When a task fails, the orchestrator does not panic and it does not blindly retry. Like a hospital triage desk, it first asks: what kind of injury is this? The answer decides the treatment — defined in code/recovery.py:

⚡

Transient → skip

A network hiccup or a busy provider. The gateway already retried; re-planning would not help. Wave the branch off and move on.

📝

Validation → skip

A skill emitted a malformed plan — that is a bug in a prompt file. Fix the prompt, not the run. Retrying would produce the same garbage.

🛠️

Upstream → replan

The task genuinely could not get what it needed. Call the Planner back with a failure report and let it draw a detour.

One exception: if the Planner itself fails, the system never asks the Planner to fix the Planner — that way lies an infinite loop. It skips instead.

How the diagnosis is made

Triage works by reading the error message for symptoms. Here is the “transient injury” checklist:

CODE — code/recovery.py 

transient_markers = (
    "503", "502", "504",
    "timeout", "timed out",
    "connection", "connectionerror", "httpstatuserror",
    "service unavailable", "bad gateway", "gateway timeout",
)
if any(m in e for m in transient_markers):
    return "transient"
return "upstream_failure"
PLAIN ENGLISH

Here is the list of phrases that mean “the internet had a moment”...

...server-busy codes like 503...

...things that gave up waiting...

...and connections that dropped.

If the error message contains any of these symptoms...

...diagnose it as a passing condition.

Anything else: a real failure worth re-planning around.

💡
Recovery without amnesia

When the Planner is recalled, it is handed the IDs of every task that already succeeded — so the detour reuses finished work instead of redoing it. When you ask an AI assistant to add retry logic anywhere, ask for this too: “on retry, reuse completed results.”

The Critic gets two strikes, not infinite

A Critic rejecting some output is its own kind of failure — handled with a strict allowance:

1
Strike one: verdict = fail

The downstream task is marked “skipped” so it never consumes the rejected output, and a recovery Planner is spliced in to redo the branch.

2
Strike two: the redo fails too

Each branch gets exactly one recovery. A second rejection hits the cap: the branch is abandoned.

3
And it says so out loud

The final log prints a warning naming every capped branch — so a thin answer is never a silent mystery.

⚖️
Bounded retries are a design choice

Unlimited retries plus an AI that keeps failing the same way equals an infinite money fire. Capping at one recovery per branch — and reporting the cap loudly — trades a perfect answer for a predictable, debuggable system.

The black box: every run leaves a flight record

Everything a run does is written to disk as it happens — like an aircraft’s black box. Each session folder holds:

code/state/sessions/<session-id>/ one folder per run
query.txt the goal you typed, word for word
graph.json the full task graph and each node’s status
nodes/ per-task records: the EXACT prompt sent and result received
browser/ per-turn screenshots whenever the Browser skill ran

Two commands read the box. flow.py --resume <session-id> restarts an interrupted run — finished tasks stay finished, only unfinished ones rerun. And replay.py <session-id> walks the run node by node, showing each task’s skill, provider, timing, and — on demand — the full prompt it was sent.

💡
Observability is a feature you can ask for

The single best debugging question for any AI system is “what exactly was the model shown?” This codebase can always answer it, because it saves every rendered prompt. When steering an AI assistant on your own project, ask for the same: “log the exact prompt and response for every model call.”
