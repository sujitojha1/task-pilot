When Things Break

Failures are not exceptions here — they are inputs. The system triages every one and keeps a black box of the whole flight.

The triage desk

When a task fails, the orchestrator does not panic and it does not blindly retry. Like a hospital triage desk, it first asks: what kind of injury is this? The answer decides the treatment — defined in code/recovery.py:

⚡

Transient → skip

A network hiccup or a busy provider. The gateway already retried; re-planning would not help. Wave the branch off and move on.

📝

Validation → skip

A skill emitted a malformed plan — that is a bug in a prompt file. Fix the prompt, not the run. Retrying would produce the same garbage.

🛠️

Upstream → replan

The task genuinely could not get what it needed. Call the Planner back with a failure report and let it draw a detour.

One exception: if the Planner itself fails, the system never asks the Planner to fix the Planner — that way lies an infinite loop. It skips instead.

How the diagnosis is made

Triage works by reading the error message for symptoms. Here is the “transient injury” checklist:

CODE — code/recovery.py 

transient_markers = (
    "503", "502", "504",
    "timeout", "timed out",
    "connection", "connectionerror", "httpstatuserror",
    "service unavailable", "bad gateway", "gateway timeout",
)
if any(m in e for m in transient_markers):
    return "transient"
return "upstream_failure"
PLAIN ENGLISH

Here is the list of phrases that mean “the internet had a moment”...

...server-busy codes like 503...

...things that gave up waiting...

...and connections that dropped.

If the error message contains any of these symptoms...

...diagnose it as a passing condition.

Anything else: a real failure worth re-planning around.

💡
Recovery without amnesia

When the Planner is recalled, it is handed the IDs of every task that already succeeded — so the detour reuses finished work instead of redoing it. When you ask an AI assistant to add retry logic anywhere, ask for this too: “on retry, reuse completed results.”

The Critic gets two strikes, not infinite

A Critic rejecting some output is its own kind of failure — handled with a strict allowance:

1
Strike one: verdict = fail

The downstream task is marked “skipped” so it never consumes the rejected output, and a recovery Planner is spliced in to redo the branch.

2
Strike two: the redo fails too

Each branch gets exactly one recovery. A second rejection hits the cap: the branch is abandoned.

3
And it says so out loud

The final log prints a warning naming every capped branch — so a thin answer is never a silent mystery.

⚖️
Bounded retries are a design choice

Unlimited retries plus an AI that keeps failing the same way equals an infinite money fire. Capping at one recovery per branch — and reporting the cap loudly — trades a perfect answer for a predictable, debuggable system.

The black box: every run leaves a flight record

Everything a run does is written to disk as it happens — like an aircraft’s black box. Each session folder holds:

code/state/sessions/<session-id>/ one folder per run
query.txt the goal you typed, word for word
graph.json the full task graph and each node’s status
nodes/ per-task records: the EXACT prompt sent and result received
browser/ per-turn screenshots whenever the Browser skill ran

Two commands read the box. flow.py --resume <session-id> restarts an interrupted run — finished tasks stay finished, only unfinished ones rerun. And replay.py <session-id> walks the run node by node, showing each task’s skill, provider, timing, and — on demand — the full prompt it was sent.

💡
Observability is a feature you can ask for

The single best debugging question for any AI system is “what exactly was the model shown?” This codebase can always answer it, because it saves every rendered prompt. When steering an AI assistant on your own project, ask for the same: “log the exact prompt and response for every model call.”
