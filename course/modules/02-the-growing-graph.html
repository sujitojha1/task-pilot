The Growing Graph

The plan is not written once and executed. It is a living map that draws new roads while the agent is already driving.

A detective’s evidence board

Picture a detective’s evidence board: photos pinned to a wall, strings connecting them. When a new lead comes in, the detective pins a new photo and runs new strings to it. The board grows as the case unfolds — nobody knew its final shape on day one.

That board is the agent’s plan. Each pinned photo is a node (one task, run by one skill), and the strings are arrows saying “this task feeds that one.” Engineers call this shape a DAG. The orchestrator in code/flow.py is the detective: it never investigates anything itself, it just decides which leads are ready to chase and pins what comes back.

🧩
Why a graph and not a checklist?

A checklist runs one item at a time. A graph knows which tasks don’t depend on each other — so three research tasks can run at the same time, and the comparison task waits for all three. Same work, much faster.

The heartbeat: run what’s ready, repeat

The whole engine is one loop. Every beat, it asks: which tasks have all their inputs finished? Run those. Pin the results. Ask again.

CODE — code/flow.py 

while True:
    ready = graph.ready_nodes()
    if not ready and not graph.has_running():
        break
PLAIN ENGLISH

Keep going around this loop forever (until told to stop)...

Ask the board: which tasks have every upstream task finished?

If nothing is ready AND nothing is still running...

...the case is closed. Stop the loop.

And here is the rule that decides “ready” — a task can start when every task feeding it is either done or deliberately skipped:

CODE — code/flow.py 

preds = list(self.g.predecessors(nid))
if all(self.g.nodes[p]["status"] in ("complete", "skipped") for p in preds):
    out.append(nid)
PLAIN ENGLISH

Gather every task whose string points INTO this one — its suppliers.

If every single supplier has either finished or been waved off...

...this task goes on the “run it now” list.

Five ways new pins land on the board

The graph starts with a single node — the Planner — and grows through exactly five mechanisms:

🌱

The Planner’s seed plan

The Planner reads your goal and pins the first batch of tasks — the opening leads of the case.

➕

Dynamic successors

Any skill can finish its work and say “now someone should do THIS next” — pinning follow-up tasks itself.

🔗

Static successors

Some pairs are wired in config: whenever the Coder finishes, a sandbox run is automatically pinned after it.

🔍

Critic auto-insertion

Skills marked critic: true get a quality-check node spliced onto every outgoing string before anyone uses their output.

🛠️

Recovery re-planning

When a task fails, the Planner is called back with a failure report to draw a detour around the dead end.

💡
Growth needs a leash

A plan that can extend itself can also run away with itself. One line guards the whole system: MAX_NODES = 60 — a hard cap so a looping Planner cannot grow the board forever. Whenever you build something self-extending, give it a leash first.
